# 检查并修复新文章提示、筛选和分页逻辑

## Goal

检查并修复文章列表在后台自动刷新、新文章提示、全部/未读筛选和无限分页之间的状态联动。用户挂机期间，新文章只能进入查询缓存并通过提示条告知，不能直接插入当前可见列表；用户点击提示条后，应显示最新文章并将列表重置到第一页和顶部。

## What I already know

- 用户明确要求：后台自动刷新发现新文章时，当前前端列表保持不变；点击“有 X 篇新文章，点击查看”后才显示新文章；点击后应保持在第一页。
- 文章列表使用 TanStack Query `useInfiniteQuery`，每 2 分钟自动 refetch。
- 全部、未读、收藏通过包含筛选参数的独立 query key 隔离缓存。
- 后端使用按 `published_at`、`created_at`、`id` 排序的 opaque cursor，新增顶部文章不会破坏已取得的历史游标。
- 当前新文章识别只比较第一页未知 ID，并自动确认新增的历史页 ID，避免加载下一页误报新文章。
- 当前点击提示条只确认所有已缓存 ID、清空计数并滚到索引 0，没有裁掉已加载的后续页，因此并未真正回到“第一页”状态。
- 当前提示条只在 `articles.length > 0` 的渲染分支显示；如果旧列表为空而后台获得了被隐藏的新文章，页面会显示“暂无文章”且用户无法点击提示条。

## Assumptions

- “保持在第一页”指点击提示条后 React Query 中当前筛选的 infinite query 仅保留第一页及其初始 `pageParam`，后续滚动再从第一页的 `next_cursor` 继续加载。
- 点击提示条展示后台轮询已经取回的最新第一页，不额外发起重复网络请求。
- 新文章提示状态按 feed 和筛选维度重置；全部和未读查询互不共享确认基线。
- 本任务不改变后端文章排序、游标格式或自动刷新间隔。

## Open Questions

- 无。用户已明确确认目标交互，技术细节可由现有代码和规范推导。

## Requirements

### 1. 后台自动刷新

- 自动 refetch 取得的新文章不得在用户点击提示条前进入可见文章数组。
- 提示计数只统计第一页中尚未确认的新文章，不得把无限滚动加载的历史文章计为新文章。
- 后台 refetch 不得改变用户当前滚动位置。

### 2. 点击新文章提示

- 点击后立即显示当前缓存中的最新第一页文章。
- 当前筛选查询已加载的第二页及后续页必须从缓存中移除，`pageParams` 同步重置到初始第一页参数。
- 列表滚动到顶部，下一次触底从最新第一页返回的 `next_cursor` 继续加载。
- 提示计数清零，已展示的第一页文章成为新的确认基线。

### 3. 全部/未读筛选与空列表

- 全部、未读筛选继续使用各自的 query key、确认基线和分页链。
- 切换筛选或 feed 时清空旧提示状态，并以新查询首次成功数据作为基线。
- 即使点击前可见旧文章为 0，只要检测到新文章，提示条仍必须显示并可点击。
- 没有可见文章且没有新文章时才显示“暂无文章”。

### 4. 分页一致性

- 点击提示后 `hasNextPage` 必须基于保留的最新第一页计算。
- 扁平列表继续按文章 ID 去重。
- 不改变现有游标分页和 read/star mutation 的稳定展示语义。

## Acceptance Criteria

- [x] 初始加载后滚动到第二页，后台 refetch 发现 N 篇新文章时，当前可见列表和滚动位置不变，并显示 N 篇新文章提示。
- [x] 点击提示后，新文章显示在顶部，列表滚动到顶部，当前 infinite query 只保留第一页。
- [x] 点击后再次触底，按最新第一页的 `next_cursor` 加载第二页，不跳页、不重复。
- [x] 在全部和未读筛选中分别执行上述流程，查询状态互不污染。
- [x] 旧可见列表为空但后台发现新文章时，提示条仍显示；点击后文章正常出现。
- [x] 初次加载和普通加载下一页不会产生假提示。
- [x] 为第一页重置和新文章确认逻辑补充自动化测试。
- [x] `npm run test`、`npm run lint`、`npm run build` 通过。

## Technical Notes

- 主要文件：`frontend/src/components/ArticleList.tsx`、`frontend/src/api/hooks.ts`、`frontend/src/lib/articleList.ts`、`frontend/src/lib/articleList.test.ts`。
- React Query 服务端状态必须留在查询缓存，不放入 Zustand。
- 重置 infinite query 时必须同时裁剪 `pages` 和 `pageParams`，避免 UI 页数与后续分页参数不一致。
- 新文章确认协调使用 `useLayoutEffect`，确保未知文章在浏览器绘制前被隐藏，避免列表闪现和滚动扰动。
- 确认 ID 集合扩展时同步更新只读状态快照，确保提示数量不变时追加的历史页仍会重新计算可见列表。
- 研究记录见 `research/current-behavior.md`。
