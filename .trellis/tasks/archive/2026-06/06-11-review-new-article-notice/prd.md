# 修复新文章提示逻辑

## Goal

修复“有 X 篇新文章，点击查看”功能中已确认的全部缺陷。后台检测不得替换当前可见列表；新文章数量必须由服务端按抓取时间和当前筛选准确计算；用户点击后才刷新到最新第一页并重建分页。

## What I already know

* 用户已观察到该功能存在多处问题，尤其是提示出现时机不正确。
* 当前分支最近包含两次相关修复：新文章刷新后的空列表，以及刷新分页稳定性。
* 当前实现以“第一页未知 ID”作为新文章判据，并直接从刷新后的 React Query 分页渲染列表。
* 用户要求修复审查中确认的全部问题。

## Assumptions (temporary)

* 保持现有两分钟自动检测间隔。
* 不改变文章列表按发布时间排序的产品行为。
* 新文章定义为基线建立后新入库且匹配当前 feed/未读/收藏筛选的文章。

## Open Questions

* 无阻塞问题。

## Requirements (evolving)

### 服务端基线与计数

* 文章列表第一页响应提供稳定的服务端基线时间。
* 新文章数量由独立接口按 `created_at > baseline` 统计，不以客户端是否见过 ID 判断。
* 计数应用与文章列表相同的 feed、未读、收藏筛选。
* 计数覆盖全部匹配文章，不受第一页 50 篇限制，也能统计发布时间较早但刚抓取入库的文章。

### 列表快照与点击行为

* 自动检测只更新提示数量，不重取或替换当前 infinite query，点击前文章和滚动位置保持不变。
* 点击提示后裁剪当前查询到第一页，重新请求最新第一页，更新基线并清空提示。
* `pages` 与 `pageParams` 同步重建；后续分页从最新第一页游标继续。
* 点击后的 Virtuoso 顶部重置必须在新第一页提交后执行。

### 刷新与初始化

* 手动刷新在 feed 抓取和提示计数重取完成前保持 pending 状态。
* 快捷键、命令面板和单 feed 刷新不得绕过提示直接替换当前列表。
* 新增/导入订阅的初始抓取不作为需要用户确认的新文章；空列表在订阅首次初始化完成后直接建立新基线并加载首批文章。

### 测试

* 覆盖服务端基线计数的筛选、超过一页和回填发布时间场景。
* 覆盖前端点击前快照不变、点击后第一页重建和刷新 pending 时序。

## Acceptance Criteria (evolving)

* [x] 自动检测发现新文章时，已加载列表内容和滚动位置完全不变。
* [x] 未读/收藏成员变化不会把历史文章误报为新文章。
* [x] 提示数量等于服务端基线后全部匹配文章数，不受分页限制。
* [x] 发布时间较早但刚入库的文章能够计入提示。
* [x] 点击提示后显示最新第一页，分页链和滚动位置正确重置。
* [x] 手动刷新按钮保持 pending，直到提示数量完成更新。
* [x] 新增/导入订阅的首批文章直接显示，不出现初始化历史提示。
* [x] 前后端相关测试、lint、typecheck/build 通过。

## Definition of Done (team quality bar)

* 完成前后端实现和自动化测试。
* 前端 lint、测试、构建通过；后端相关测试通过。
* 更新从本次修复中得到的项目规范。

## Out of Scope (explicit)

* 不进行无关重构或样式调整。
* 不修改自动抓取调度周期。
* 不改变文章排序规则。
* 不引入推送、SSE 或 WebSocket。

## Technical Notes

* 审查记录：`research/current-behavior.md`。
* 已确认六类问题：点击前列表被替换、筛选变化误报、计数与隐藏范围不一致、第一页上限/回填文章漏报、手动刷新完成时序错误、新订阅后台抓取通知延迟。
* 技术决策：移除客户端 ID 确认集合；使用服务端基线时间 + 独立 count query；infinite query 不再定时 refetch。
* 初始化抓取由 `initial_fetch_pending` 驱动 2 秒轮询，整批结束后只刷新一次；失败抓取不保持 pending。
